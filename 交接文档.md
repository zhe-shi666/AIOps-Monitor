> 交接日期：2026-04-27

## 一、项目概述

**AIOps-Monitor** 是一个基于 AI 的智能运维监控系统，支持单机和分布式两种部署模式。

核心功能：
- 实时采集本机 CPU / 内存指标，通过 WebSocket 推送到前端图表
- 阈值告警（CPU > 70%，内存 > 90%）触发 AI 自动诊断，生成 Markdown 格式的根因分析报告
- 用户注册/登录（JWT 鉴权），ADMIN 角色可管理用户
- 支持单机模式（standalone）和分布式集群模式（distributed），通过配置文件切换

## 四、数据库表结构

执行 `mysql.sql` 初始化，共 4 张表：

| 表名 | 说明 |
|------|------|
| `users` | 用户表，角色 ADMIN / USER，第一个注册用户自动成为 ADMIN |
| `monitor_targets` | 监控目标（Agent 上报身份凭证），当前后端未完全实现 |
| `system_metrics_history` | CPU/内存历史数据，含 `user_id`、`target_id` 字段（待关联） |
| `incident_log` | 告警记录，含 `user_id`、`target_id` 字段（待关联） |

> 注意：`mysql.sql` 中 `incident_log` 和 `system_metrics_history` 出现了两次定义（旧版无 user_id/target_id，新版有），以**文件靠前的新版**为准，执行时会自动 DROP 旧表。

## 五、本地启动步骤

### 前置依赖

- JDK 21
- Maven 3.9+
- MySQL 8.0（端口 3307，库名 `aiops_monitor`，用户 root/root）
- Redis（端口 6379，无密码）
- Ollama（本地运行，端口 11434，已拉取 `qwen3:0.6b` 模型）
- Node.js 18+

### 后端启动

```bash
cd ai-ops-monitor

# 开发模式（standalone，连本地 MySQL 3307）
./mvnw spring-boot:run -Dspring-boot.run.profiles=dev

# 或生产/集群模式
./mvnw spring-boot:run -Dspring-boot.run.profiles=prod
```

服务启动后监听 `http://localhost:8080`

### 前端启动

```bash
cd aiops-vue
npm install
npm run dev
```

前端默认运行在 `http://localhost:5173`

### Ollama 模型

```bash
# 拉取模型（首次需要）
ollama pull qwen3:0.6b

# 启动 Ollama 服务
ollama serve
```

## 六、关键配置说明

### application.yml（公共）

```yaml
jwt:
  secret: aiops-monitor-jwt-secret-key-2026-change-in-production  # 生产环境必须替换
  expiration: 86400000  # Token 有效期 24 小时

spring:
  ai:
    ollama:
      chat:
        options:
          model: qwen3:0.6b  # 可替换为其他 Ollama 模型
```

### 切换单机/集群模式

- `application-dev.yml`：`monitor.mode: standalone`
- `application-prod.yml`：`monitor.mode: distributed`

分布式模式下，多个后端节点共享同一 MySQL 和 Redis，Redis 用于分布式锁（防止多节点重复触发 AI 诊断）。

## 七、核心业务流程

```
[定时任务 5s] CollectorScheduler
    → SystemHardwareCollector 采集 CPU/内存
    → MetricsPublisher 发布指标到 WebSocket /topic/metrics
    → AlarmCheckTask 检查阈值
        → 超阈值 → 写 incident_log
        → 调用 AiService.getDiagnosticReportAsync()（异步，30s 冷却）
            → Ollama AI 生成诊断报告
            → 推送到 WebSocket /topic/ai-reports

[前端 Dashboard]
    → SockJS + STOMP 连接 ws://localhost:8080/ws-monitor
    → 订阅 /topic/metrics → ECharts 实时折线图
    → 订阅 /topic/ai-reports → Markdown 渲染报告列表
```

## 八、API 接口一览

### 认证（无需 Token）

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/auth/login` | 登录，返回 JWT token |
| POST | `/api/auth/register` | 注册，首个用户自动为 ADMIN |

### 管理后台（需 ADMIN 角色）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/admin/users` | 分页查询用户列表，支持 username 模糊搜索 |
| GET | `/api/admin/users/{id}` | 用户详情 |
| PUT | `/api/admin/users/{id}/role` | 修改用户角色 |
| PUT | `/api/admin/users/{id}/status` | 启用/禁用用户 |
| DELETE | `/api/admin/users/{id}` | 删除用户 |
| GET | `/api/admin/stats` | 统计数据（用户数、今日告警数） |

### 监控指标（Prometheus）

- `GET /actuator/prometheus` — Prometheus 格式指标
- `GET /actuator/health` — 健康检查

## 十、待完善的功能（已知 TODO）

以下功能在数据库和部分后端代码中已有结构，但尚未完整实现：

1. **monitor_targets 表**：Agent 多目标监控的注册与管理接口未实现，`system_metrics_history` 和 `incident_log` 中的 `target_id` / `user_id` 字段目前未被写入。

2. **Dashboard 硬件信息**：前端硬件信息卡片（CPU 型号、内存总量）目前是硬编码的假数据（`Intel Core i7-12700K / 32GB DDR4`），需要后端提供接口返回真实数据。

3. **告警阈值可配置化**：CPU 阈值（70%）和内存阈值（90%）目前硬编码在 `AlarmCheckTask.java`，可以做成数据库配置或接口动态调整。

4. **WebSocket 鉴权**：`WebSocketAuthInterceptor` 已存在，但需要验证 Token 校验逻辑是否完整覆盖所有订阅频道。

5. **前端 API 地址硬编码**：`DashboardView.vue` 中 WebSocket 地址写死为 `http://localhost:8080`，生产部署需要改为环境变量配置。

6. **分布式模式下的节点注册**：集群模式下各节点通过 `spring.application.name` 区分，但没有节点注册/下线的生命周期管理。

## 十一、注意事项

- **JWT Secret**：`application.yml` 中的 secret 是示例值，上线前必须替换为强随机字符串。
- **MySQL 端口**：配置文件中使用的是 `3307`（非默认 3306），注意本地 MySQL 端口配置。
- **Spring AI 版本**：使用的是 `1.0.0-M5` 里程碑版本，依赖从 Spring Milestones 仓库拉取，网络不好时可能需要配置代理或镜像。
- **Ollama 必须先启动**：后端启动时会尝试连接 Ollama，如果 Ollama 未运行，AI 诊断功能会报错但不影响监控主流程。
