# Phase-4 交付与验收说明

更新时间：2026-04-28

## 1. 交付范围

1. 动作治理审计  
- 新增 `ai_action_audit`，记录动作创建、审批、执行、重试、回滚演练、回滚执行等事件。  
- 每条记录保留决策结果、风险等级、操作者和结构化细节。

2. 风险分级审批策略  
- `LOW`：可按请求选择是否审批。  
- `MEDIUM/HIGH`：强制审批。  
- `HIGH`：动作创建时要求 `rollbackPlan`，审批时要求 `note`。

3. 回滚执行闭环  
- 新增 `ai_rollback_run`，支持回滚演练与正式回滚。  
- 正式回滚成功后动作状态置为 `ROLLED_BACK`。  
- 回滚事件同步进入 Timeline 与审计记录。

4. 前端治理工作台  
- AI 专家新增「治理 Governance」Tab。  
- 展示动作审计流与回滚记录。  
- 在动作卡片支持「回滚演练」「执行回滚」。

## 3. 新增/增强接口

1. 治理查询  
- `GET /api/investigations/{id}/actions/audits`  
- `GET /api/investigations/{id}/actions/{actionId}/rollback-runs`

2. 回滚操作  
- `POST /api/investigations/{id}/actions/{actionId}/rollback-drill`  
- `POST /api/investigations/{id}/actions/{actionId}/rollback-execute`

3. 调查详情增强  
- `GET /api/investigations/{id}` 返回 `rollbackRuns`、`recentActionAudits`。

## 4. Phase-4 Exit Criteria 对应

1. Runbook 执行可审计  
- 已完成：动作生命周期事件落库 `ai_action_audit`，并可分页查询。

2. 审批流按风险级别生效  
- 已完成：后端强制中高风险审批，高风险审批需审批意见。

3. 回滚流程可演练通过  
- 已完成：支持演练与正式回滚，结果落库并进入时间线。

## 5. 验证建议

1. 创建 `HIGH` 风险动作且不填 `rollbackPlan`，应返回 400。  
2. 对 `HIGH` 风险动作审批不填 `note`，应返回 400。  
3. 执行成功动作后发起回滚演练，应生成 `ai_rollback_run` 记录。  
4. 执行正式回滚成功后，动作状态应变为 `ROLLED_BACK`。  
5. 在治理 Tab 与 Timeline 可看到完整审计与回滚轨迹。
