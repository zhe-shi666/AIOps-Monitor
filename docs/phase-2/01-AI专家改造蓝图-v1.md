# AI 专家改造蓝图 v1

更新时间：2026-04-27

## 1. 目标

将 AI 专家能力从“告警后生成一段文本”升级为“可追溯、可协作、可执行”的调查中枢：

1. 统一接管告警调查流程（创建调查、收集证据、生成假设、提出动作）。
2. 支持多指标上下文（CPU/MEMORY/DISK/NET_RX/NET_TX/PROCESS_COUNT）。
3. 输出结构化结果（风险等级、根因候选、证据链、动作建议、回滚计划）。
4. 建立操作闭环（审批、执行、结果审计、复盘快照）。

## 2. 分层架构

1. 事件层（Event）
   - 输入：告警中心、Agent 心跳告警、本机巡检告警。
   - 输出：Investigation 创建请求。
2. 编排层（Orchestration）
   - 核心组件：`InvestigationOrchestrator`（后续实现）。
   - 职责：去重聚合、收集证据、调用 AI、存档结果。
3. 认知层（AI）
   - 核心组件：`AiService` + 结构化 Prompt/Schema。
   - 输出：`hypotheses + action_plan + summary`。
4. 执行层（Action）
   - 审批/执行动作，写入执行日志。
5. 展示层（UI）
   - 右侧 AI 专家区升级为“调查工作台”。

## 3. 里程碑

### Phase A（当前启动）

1. 数据模型落地（6 张表 + 实体 + Repository）。
2. 统一调查对象（Investigation）可创建与查询。
3. 结构化报告快照可持久化。

### Phase B

1. 接入告警入口到 `InvestigationOrchestrator`。
2. AI 输出改为 JSON Schema 约束。
3. WebSocket 增加 investigation 事件推送。

### Phase C

1. 前端 AI 专家区切换为“证据/假设/动作/执行”视图。
2. 动作审批与执行日志链路贯通。

### Phase D

1. 自动复盘草稿（Postmortem）生成。
2. 质量指标看板（误报率、平均定位时长、执行成功率）。

## 4. 与现状差距（基于代码）

1. `PromptDataBuilder` 仍以 CPU/MEMORY 为主，缺少全量指标证据拼接。
2. `AlarmCheckTask` 仅触发 CPU/MEMORY AI 诊断。
3. `AgentIngestController` 虽有多指标告警，但未接入调查对象模型。
4. 前端 `AiExpertPanel` 仅是报告流，缺乏调查状态机和动作操作面。

## 5. 执行原则

1. 不回头重写：采用“兼容迁移”，保留旧 `/topic/ai-reports`，逐步切到 investigation 频道。
2. 最小风险上线：先加表和对象，不破坏现有告警中心逻辑。
3. 多租户优先：所有 investigation 相关对象必须带 `user_id`。
