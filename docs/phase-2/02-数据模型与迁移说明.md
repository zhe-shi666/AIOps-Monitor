# Phase-2 数据模型与迁移说明

更新时间：2026-04-27

## 1. 新增对象

1. `ai_investigation`
   - 调查主对象，承载状态、等级、摘要、置信度与生命周期。
2. `ai_observation`
   - 证据对象，记录指标/日志/链路/变更等观测点。
3. `ai_hypothesis`
   - 根因假设对象，保存候选根因、排序与置信度。
4. `ai_action_plan`
   - 动作建议对象，保存命令/runbook、风险级别、审批要求。
5. `ai_action_run`
   - 动作执行对象，保存执行过程与结果审计。
6. `ai_report_snapshot`
   - 报告快照对象，存储 Markdown 与 JSON 双格式版本。

## 2. 迁移脚本

脚本文件：

`mysql/migrations/2026-04-27_phase2_ai_investigation.sql`

执行顺序（建议）：

1. 先执行所有 phase-1 迁移。
2. 再执行 phase-2 迁移。
3. 启动后端，确认 JPA 未触发异常结构变更。

## 3. 联调检查项

1. 表结构检查：
   - `SHOW TABLES LIKE 'ai_%';`
2. 约束检查：
   - `ai_investigation.user_id -> users.id`
   - `ai_investigation.incident_id -> incident_log.id`
   - `ai_investigation.target_id -> monitor_targets.id`
3. 索引检查：
   - investigation 状态检索索引
   - observation 按 investigation + time 查询索引
   - action_run 按 action_plan + status 查询索引

## 4. 已落地接口（Phase A）

1. `POST /api/investigations`
2. `GET /api/investigations/{id}`
3. `GET /api/investigations`（分页）
4. `GET /api/investigations/{id}/timeline`
5. `POST /api/investigations/{id}/close`
6. `POST /api/investigations/{id}/actions`
7. `POST /api/investigations/{id}/actions/{actionId}/approve`
8. `POST /api/investigations/{id}/actions/{actionId}/execute`
9. `POST /api/investigations/{id}/snapshots`

## 5. 已接入链路（Phase A）

1. Agent 告警落库后会自动创建 Investigation（`IncidentLog -> AiInvestigation`）。
2. 创建时自动写入首条 Observation（告警指标证据）。
3. 创建时自动写入第 1 版 Report Snapshot（系统占位报告）。

## 6. 下一步接口落地建议

1. `POST /api/investigations/{id}/hypotheses`（写入根因假设）
2. `POST /api/investigations/{id}/observations`（追加证据）
3. `POST /api/investigations/{id}/actions/{actionId}/retry`（显式重试语义）
