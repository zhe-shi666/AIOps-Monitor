# AIOps-Monitor 里程碑与 DoD（V1 冻结版）

> 版本：v1  
> 日期：2026-04-27

## 1. 总体节奏（固定）

1. Phase 0：2026-04-27 ~ 2026-05-08
- 交付：PRD、架构、数据字典、里程碑与规范（冻结）

2. Phase 1：2026-05-11 ~ 2026-06-12
- 交付：主机监控基座（资产、采集、查询、规则告警、通知）

3. Phase 2：2026-06-15 ~ 2026-07-24
- 交付：日志接入、链路接入、事件中心（聚合/去重/抑制）

4. Phase 3：2026-07-27 ~ 2026-08-28
- 交付：AIOps 引擎（异常检测、关联分析、RCA 候选）

5. Phase 4：2026-08-31 ~ 2026-09-25
- 交付：Runbook、审批流、执行审计、回滚机制

6. Phase 5：2026-09-28 ~ 2026-10-23
- 交付：企业级前端工作台与设计系统收敛

7. Phase 6：2026-10-26 ~ 2026-11-20
- 交付：压测、演练、灰度、上线、运维交接

## 2. 每阶段通用 DoD（必须全部满足）

1. 功能
- 需求范围内功能可演示、可验收、可回归。

2. 质量
- 单元测试/集成测试通过。
- 核心链路有监控与告警。

3. 安全
- 权限校验完整。
- 高危操作有保护措施。

4. 运维
- 有部署说明、回滚说明、故障处理手册。

5. 文档
- API 文档、变更记录、已知限制完整。

## 3. 阶段退出门槛（Exit Criteria）

### 3.1 Phase 1 Exit

1. 主机采集覆盖 CPU/内存/磁盘/网络/进程。
2. 告警通知链路全通。
3. 多租户隔离可验证。

### 3.2 Phase 2 Exit

1. 日志与链路能关联到主机/服务维度。
2. 事件中心具备聚合与去重能力。
3. 告警噪声明显下降（可量化）。

### 3.3 Phase 3 Exit

1. 异常检测可稳定输出。
2. RCA 报告包含证据链与置信度。
3. 可追踪模型输入输出。

### 3.4 Phase 4 Exit

1. Runbook 执行可审计。
2. 审批流按风险级别生效。
3. 回滚流程可演练通过。

### 3.5 Phase 5 Exit

1. 设计系统在核心页面统一生效。
2. 事件工作台支持全流程闭环。
3. 前端性能满足目标（关键页面可用）。

### 3.6 Phase 6 Exit

1. 压测达标。
2. 故障演练通过。
3. 上线方案、回滚方案、交接文档齐全。

## 4. GitHub Desktop 执行规范（固定）

1. 分支策略
- `main`：生产
- `develop`：集成
- `feature/*`：功能开发
- `release/*`：发布准备
- `hotfix/*`：线上紧急修复

2. 提交流程
- Issue 建立 -> 新建 feature 分支 -> 本地开发 -> PR -> 评审 -> 合并

3. 质量闸门
- CI 全绿才可合并。
- 至少 1 位评审通过。
- 关键模块变更需补充测试。

4. 提交规范
- `feat:` `fix:` `refactor:` `docs:` `chore:`
- 提交描述必须说明影响范围。

## 5. 变更控制（防回头）

1. 当前阶段只接 P0/P1 变更。
2. 新需求默认放下一阶段。
3. 架构级变更需 ADR。
4. 每周固定一次变更评审会。

## 6. 风险与预案

1. 风险：租户规模不确定导致容量突增。
- 预案：采用 L1/L2/L3 分层容量方案，按层扩容。

2. 风险：告警噪声过高影响可用性。
- 预案：优先建设聚合、抑制、静默策略。

3. 风险：AI 建议准确性波动。
- 预案：保留人工确认环节，记录反馈持续优化。

4. 风险：需求漂移导致反复返工。
- 预案：严格执行本文档冻结和变更单机制。
