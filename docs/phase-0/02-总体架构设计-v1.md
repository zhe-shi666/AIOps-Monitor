# AIOps-Monitor 总体架构设计（V1 冻结版）

> 版本：v1  
> 日期：2026-04-27

## 1. 架构目标

1. 面向多租户与可扩展容量，优先适配虚机/物理机环境。
2. 后端保持模块化、边界清晰、可测试，避免过早微服务化。
3. 支持从当前实现平滑演进，不推翻重来。

## 2. 分层架构

1. 接入层
- Web 前端（Vue）
- Open API（REST/WebSocket）
- Agent 接入网关（主机数据上报）

2. 控制层
- 身份与权限（JWT -> 可扩展 OIDC）
- 租户与配额管理
- 资产管理与注册中心
- 告警策略管理

3. 数据处理层
- 指标处理（标准化、清洗、聚合）
- 事件处理（去重、聚合、抑制、静默）
- AI 分析引擎（异常检测、RCA 候选）

4. 数据存储层
- 关系型存储（MySQL）：元数据、配置、审计、事件索引
- 时序存储（阶段化引入）：指标样本
- 缓存与分布式协调（Redis）：热点数据、锁、消息分发
- 对象存储（阶段化引入）：报告与归档

5. 通知与执行层
- 通知通道（企业微信、邮件、Webhook）
- Runbook 执行器（审批 + 记录 + 回滚）

## 3. 逻辑模块边界（后端）

1. `identity`：认证、授权、Token、租户上下文。
2. `asset`：主机资产、Agent 注册、状态管理。
3. `ingest`：指标/日志/事件接入与标准化。
4. `monitor`：查询、聚合、实时推送。
5. `alert`：规则引擎、事件中心、通知编排。
6. `aiops`：异常检测、事件关联、RCA 建议。
7. `automation`：Runbook、审批流、执行日志。
8. `audit`：操作审计、变更追踪、合规导出。

说明：当前可保留单体部署，但代码必须按模块目录与接口边界组织，未来可按模块拆分服务。

## 4. 关键数据流

### 4.1 指标链路

1. Agent 上报主机指标。
2. 接入层鉴权并写入处理队列/处理器。
3. 标准化（tenant_id、host_id、metric_key、tags、timestamp）。
4. 写入时序存储并推送实时通道。
5. 告警引擎消费指标进行规则检测与异常检测。

### 4.2 事件链路

1. 告警或系统事件进入事件中心。
2. 事件中心执行去重、分组、抑制、静默策略。
3. 输出统一事件对象（Incident/Event）。
4. 通知中心按升级策略发送通知。
5. AIOps 进行关联分析并补充 RCA 信息。

### 4.3 处置链路

1. 运维人员在事件工作台触发 Runbook。
2. 根据风险级别进入审批或直执行。
3. 执行结果、日志、耗时写入审计系统。
4. 必要时执行回滚动作并生成回滚报告。

## 5. 虚机/物理机场景部署拓扑（优先）

1. 控制面
- `api-server`（Spring Boot）
- `worker`（异步任务/AIOps/通知）
- `frontend`（Nginx + Vue）

2. 数据面
- `mysql`（元数据）
- `redis`（缓存+协调）
- `timeseries`（阶段 2 引入，可先由 MySQL 过渡）

3. 采集面
- 每台主机部署 `agent`，按租户 token 上报。
- 支持心跳、断连重试、缓冲重传。

## 6. 前端架构原则（高级优美）

1. 设计系统优先
- 颜色变量、字体层级、间距体系、组件状态统一

2. 页面结构
- 首页总览（租户全局）
- 资产视图（主机维度）
- 事件工作台（告警全流程）
- RCA 详情（证据链 + 建议）
- 运维审计与报表

3. 交互准则
- 实时刷新无明显抖动
- 高危操作二次确认
- 关键状态可追溯（时间线）

## 7. 安全与合规

1. 全请求带租户上下文。
2. 所有写操作必须记录审计日志。
3. 敏感字段（token、邮箱、手机号、IP 部分字段）脱敏展示。
4. 高风险操作（批量处置、删除策略）必须审批。

## 8. 演进路线（不推翻）

1. 当前阶段：模块化单体 + 明确接口边界。
2. 中期阶段：将 `ingest/alert/aiops` 拆分为独立进程。
3. 远期阶段：根据流量拆分微服务与独立存储。

## 9. 架构冻结规则

1. 本架构文档为 Phase 1~2 的执行基线。
2. 新增能力优先通过扩展点实现，不修改核心主干。
3. 架构级变更必须补充 ADR 后再进入开发。
